# Next Ads Offline Candidate Similarity Diagnostics

Azure Boards story: 5149979
Status: follow-up story concept, not part of the current feature-store production model contracts.

## Purpose

Candidate similarity should be treated as an offline diagnostic capability. It should help analysts understand whether existing Next Ads candidate adverts look similar to a customer's product interests, but it must not feed production pCTR, Theme Affinity, LTR, scoring, ranking, assignment, or compatibility-view outputs in this feature-store slice.

This keeps the feature-store foundation focused on governed reusable tables while leaving similarity as a separately scoped experiment/analysis route.

## Non-Production Boundary

The diagnostics output must not be read by:

- `next_uk_nextads_fs_pctr_model_input`
- `next_uk_nextads_fs_theme_affinity_model_input`
- `next_uk_nextads_pctr_features_latest`
- `next_uk_nextads_theme_affinity_features_latest`
- production scoring, ranking, assignment, or build-page jobs

Any later model experiment that wants this signal must copy or join it into an experiment-specific training dataset under a separate story, with clear evidence and release approval before it can affect production.

## Candidate Source

Similarity should be calculated only after a bounded candidate set exists. The preferred first source is the existing theme-to-ad prerank output:

```text
next_uk_nextads_preranked_ads_from_themes_latest
```

This table already represents account/ad/location candidates generated by the current Next Ads process. The diagnostics job should enrich those rows; it should not generate candidates itself.

The grain should be:

```text
account_number
advert_id
location
reference_date
candidate_source
```

The row count must be bounded by the candidate source row count. A full customer-by-ad cross join is explicitly out of scope.

## Vector Dependencies

The diagnostics job needs agreed vector sources before implementation:

| Side | Candidate source |
| --- | --- |
| Customer product interest | Future account/customer product-interest vector table, or a clearly documented CWB pCTR-derived source |
| Advert product profile | `next_uk_nextads_fs_advert_product_profile_daily` |
| Product embedding lookup | `next_uk_nextads_fs_product_embeddings_latest` |

Both sides must carry compatible model/version metadata before cosine similarity is calculated.

## Proposed Diagnostics Output

The follow-up story should add a non-serving diagnostics table such as:

```text
next_uk_nextads_fs_candidate_similarity_diagnostics
```

Recommended columns:

```text
account_number
advert_id
location
reference_date
candidate_source
candidate_rank
candidate_score
customer_product_vector_model_name
customer_product_vector_model_version
advert_product_vector_model_name
advert_product_vector_model_version
customer_product_embedding_coverage
advert_product_embedding_coverage
customer_ad_product_cosine_similarity
similarity_bucket
similarity_calculated_at
created_at
```

Registry metadata should mark it as:

```text
training_safe: false
consumers: [offline_analysis]
```

## Follow-Up Acceptance Criteria

| Criterion | Required evidence |
| --- | --- |
| No production model impact | No production job, model-input table, compatibility view, scoring job, ranking job, or assignment job reads the diagnostics table. |
| Bounded candidate source | Diagnostics row count is no greater than the selected candidate source row count for the same reference date. |
| No full cross join | Implementation reads an existing candidate table and joins vectors onto it. |
| Version-safe similarity | Customer and advert vector model/version metadata are present and checked. |
| Offline-only consumers | Registry consumers are limited to `offline_analysis`. |
